1. Execution Reliability
Booking always resolves to:
BOOKED_CONFIRMED OR
FAILED
No stale CALLING / PROPOSED attempts
No orphaned proposals
No duplicate calendar events
System is deterministic across retries
2. Data Integrity & Invariants

At most one active future confirmed event per:

(app_user_id, provider_id, profile_id)

(temporary: provider + app_user_id until profiles are live)

schedule_attempts, proposals, calendar_events remain consistent
No conflicting states across tables
All writes are idempotent where required
3. Security & Access Control (REQUIRED)

All security guarantees must be enforced at the backend and database level:

Identity Enforcement

All data access scoped by:

app_user_id
No cross-user reads or joins possible

API Security

Database Protections

No ability to access another user’s data via API or DB
Token Safety
No API keys exposed client-side
No service role leakage
Vapi + Supabase usage fully server-side
4. System Observability & Recovery
Every call produces:
transcript

System can explain:
what happened

Failed executions are:
visible

⚠️ Launch Blockers

QBH must NOT launch if ANY of the following are true:

A user can access another user’s data
Booking can succeed without writing a calendar event
Booking can fail without resolving attempt state
Duplicate future bookings can exist
Any endpoint allows unscoped reads or writes
Any API key or service credential is exposed client-side
🧭 Final Principle

A system that is functionally correct but not secure is not correct.

Security is not a later phase.
It is part of system integrity.
