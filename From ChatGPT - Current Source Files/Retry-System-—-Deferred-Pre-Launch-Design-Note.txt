Retry System — Deferred Pre-Launch Design Note
Status

Deferred until the final pre-launch checkpoint.

This is important and likely required for a truly shippable real-world system, but it is being intentionally deferred to avoid scope drift during the current hardening phase. Current work remains focused on:

execution reliability
backend correctness
invariant enforcement
system trust
Why this matters

QBH now knows:

whether a booking attempt succeeded or failed
why it failed
whether the failure should block, require user input, or be retryable

That is a major step, because the roadmap requires QBH to track booking outcomes, understand failure reasons, and support a deterministic retry system.

What has already been proven

The current system can now classify failed calls into meaningful outcome buckets, including:

NOT_ACCEPTING_NEW_PATIENTS → no retry
VOICEMAIL → retryable
OFFICE_CLOSED → retryable

This means QBH now has the core interpretation layer needed for a future retry system:

raw transcript / call evidence
structured failure classification
deterministic retry intent
Why a retry system is likely necessary before true launch

Without a retry system, QBH still behaves too much like a user-triggered dashboard for some real-world failure modes.

Example:

office closed
voicemail
no answer
callback requested

In these cases, a shippable QBH should often behave like an active coordinator, not a passive tool. The roadmap explicitly says QBH should feel like it is actively handling the user’s life, should expose current/last/next action clearly, and should retry automatically when appropriate.

Product principle

OpenAI classification should inform retry behavior, but should not itself be the retry engine.

Canonical flow should be:

call evidence → structured classification → deterministic retry decision → execution

This preserves:

backend-first truth
deterministic state
auditability
no UI inference of business logic
What the future retry system should do

If implemented, the retry system should:

retry automatically when classification says retry is appropriate
remain bounded
never create duplicate bookings
remain fully backend-owned
expose system-owned next steps in the UI
preserve audit history and lineage through schedule_attempts
stay scoped by app_user_id
never expose sensitive transcript or token data to the frontend
Candidate retry buckets

Retryable:

VOICEMAIL
OFFICE_CLOSED
NO_ANSWER
BUSY_SIGNAL
CALL_DROPPED
TRANSFER_LOOP
CALLBACK_REQUESTED
NO_AVAILABILITY_GIVEN (case-dependent)

Non-retryable / blocked:

NOT_ACCEPTING_NEW_PATIENTS
WRONG_NUMBER
PROVIDER_NOT_FOUND

User-required:

CALENDAR_CONFLICT
NEEDS_REFERRAL
INSURANCE_NOT_ACCEPTED
PATIENT_INFO_REQUIRED
Candidate retry decision object

If/when implemented, the canonical backend retry decision should look roughly like:

type RetryDecision = {
  retry_allowed: boolean;
  retry_strategy:
    | "NONE"
    | "NEXT_BUSINESS_HOURS"
    | "CALLBACK_WINDOW"
    | "SOON";
  next_action:
    | "BLOCK"
    | "AUTO_RETRY"
    | "USER_REQUIRED"
    | "REVIEW_REQUIRED";
  reason: string;
  retry_after_iso: string | null;
};
Candidate retry execution behavior

Examples:

NOT_ACCEPTING_NEW_PATIENTS
retry_allowed = false
next_action = BLOCK
VOICEMAIL
retry_allowed = true
next_action = AUTO_RETRY
use callback window if available, otherwise fallback retry timing
OFFICE_CLOSED
retry_allowed = true
next_action = AUTO_RETRY
retry next business window
CALENDAR_CONFLICT
retry_allowed = false
next_action = USER_REQUIRED
Important architectural guardrails

If/when this is built, it must follow these rules:

backend-owned only
deterministic and bounded
app_user_id scoped
no raw transcript surfaced to UI
no retry logic computed in frontend
booking invariants preserved
no duplicate future confirmed events
no mutation of audit history; append-only where possible
Why it is deferred for now

This is being deferred because the current hardening phase explicitly limits scope to:

execution reliability
backend truth enforcement
identity normalization

and warns against introducing new systems or speculative rewrites without clear necessity.

Revisit trigger

Revisit this system at the final pre-launch checkpoint.

At that time, ask:

Are retryable states still user-triggered in a way that feels non-shippable?
Do voicemail / office-closed cases need system-owned retries for launch trust?
Can this be added without violating invariants or security guarantees?
Is push/update visibility sufficient to support automatic retries?
Current conclusion

Retry logic is strategically important and probably part of the true launch-quality system, but it is not the current hardening step.

For now:

keep classification
keep deterministic retry decision derivation
keep UI reflecting backend truth
defer automated retry execution until just before launch
